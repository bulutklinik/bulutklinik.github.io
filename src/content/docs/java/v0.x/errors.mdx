# Errors & responses

## The response envelope

Every API response is a JSON envelope. The SDK unwraps it for you — on success it
returns the `data` payload directly as a Jackson `JsonNode`; on failure it throws a
typed (unchecked) exception.

```jsonc
{
  "resultType": 0,            // state code (see below)
  "errorType": "validation",  // string label OR numeric code
  "errorMessage": "…",        // human-readable, localized via `lang`
  "successMessage": "…",
  "data": { /* payload */ }   // what the SDK returns on success
}
```

A call is successful only when the HTTP status is 2xx **and** `resultType == 0`.
When the envelope's `data` is absent on a successful call the SDK returns a JSON
null node (`NullNode`), never a Java `null`. The `resultType` state codes:

| Value | Name      | Meaning |
|:-----:|-----------|---------|
| `0`   | `Success` | Return `data`. |
| `1`   | `Error`   | Throw `ApiException` (or a subtype). |
| `2`   | `Logout`  | Session revoked — token store cleared, `AuthenticationException`. |
| `3`   | `Update`  | Client/app too old — `ApiException`. |
| `4`   | `Refresh` | Token expired — triggers silent refresh + retry. |

> `errorType` is **polymorphic**: some endpoints return a string label
> (`"validation"`), others a numeric code. The SDK exposes it as `Object` on the
> error context and treats it as a refinement hint only when it is a `String`.

## Exception hierarchy

All exceptions extend `BulutklinikException`, which extends `RuntimeException` (so
they are **unchecked** — no `throws` clause is required):

```
BulutklinikException              (base — all SDK exceptions)
├── TransportException            (network failure, timeout, DNS, TLS — no HTTP response)
└── ApiException                  (an HTTP response that wasn't a success)
    ├── ValidationException       (422, or errorType "validation")
    ├── AuthenticationException   (401 / logout / failed refresh)
    ├── AuthorizationException    (403)
    ├── NotFoundException         (404)
    └── RateLimitException        (429 — retry-after in the context)
```

`TransportException` carries the underlying cause via `getCause()`. Every
`ApiException` carries an `ApiErrorContext` reachable with `e.context()`, and a
convenience `e.httpStatus()` shortcut. `ApiErrorContext` is a record:

| Component       | Type       | Notes |
|-----------------|------------|-------|
| `httpStatus()`  | `int`      | The HTTP status code. |
| `resultType()`  | `Integer`  | Envelope `resultType`; may be `null`. |
| `errorType()`   | `Object`   | String label or numeric code; may be `null`. |
| `data()`        | `JsonNode` | Raw `data` from the failed envelope. |
| `method()`      | `String`   | HTTP method of the failed request. |
| `path()`        | `String`   | Path of the failed request. |
| `retryAfter()`  | `Integer`  | Seconds to wait (`429` only, when `Retry-After` is present); else `null`. |

```java
import com.bulutklinik.sdk.*;

try {
    client.payments.pay(input);
} catch (RateLimitException e) {
    System.out.println("throttled — retry after " + e.context().retryAfter() + "s");
} catch (ValidationException e) {
    System.out.println("invalid request: " + e.context().data());
} catch (AuthenticationException e) {
    System.out.println("session expired — re-login required");
} catch (TransportException e) {
    System.out.println("network problem: " + e.getCause());
} catch (ApiException e) {
    System.out.println("API error " + e.httpStatus()
            + " on " + e.context().method() + " " + e.context().path());
}
```

Because these are unchecked exceptions, the order of `catch` blocks matters: list the
more specific subtypes (e.g. `RateLimitException`, `ValidationException`) before the
broader `ApiException`, and `ApiException` before `BulutklinikException`.

## Mapping precedence

The SDK picks the most specific exception type in this order (see
`ApiException.of`):

1. `resultType == 2` (logout) → `AuthenticationException`
2. string `errorType == "validation"` (case-insensitive) **or** HTTP 422 → `ValidationException`
3. HTTP status: 401 → `AuthenticationException`, 403 → `AuthorizationException`,
   404 → `NotFoundException`, 429 → `RateLimitException`
4. otherwise (including numeric `errorType`, or 2xx with `resultType != 0`) → `ApiException`

Because `errorType` may be numeric, the SDK guards (`errorType instanceof String`)
before string-matching it — and so should any code that inspects
`e.context().errorType()`.
