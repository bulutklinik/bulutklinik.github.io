# Errors & responses

## The response envelope

Every API response is a JSON envelope. The SDK unwraps it for you — on success it
returns the `data` payload directly (as an `` `nlohmann::json` ``); on failure it
throws a typed exception.

```jsonc
{
  "resultType": 0,            // state code (see below)
  "errorType": "validation",  // string label OR numeric code
  "errorMessage": "…",        // human-readable, localized via `lang`
  "successMessage": "…",
  "data": { /* payload */ }   // what the SDK returns on success
}
```

A call is successful only when the HTTP status is 2xx **and** `resultType == 0`.
The `resultType` state codes:

| Value | Name      | Meaning |
|:-----:|-----------|---------|
| `0`   | `Success` | Return `data`. |
| `1`   | `Error`   | Throw `ApiError` (or a subtype). |
| `2`   | `Logout`  | Session revoked — token store cleared, `AuthenticationError`. |
| `3`   | `Update`  | Client/app too old — `ApiError`. |
| `4`   | `Refresh` | Token expired — triggers silent refresh + retry. |

> `errorType` is **polymorphic**: some endpoints return a string label
> (`"validation"`), others a numeric code. The SDK keeps it as a raw
> `` `nlohmann::json` `` value and treats it as a refinement hint only when it is a
> string — guard with `error_type.is_string()` before string-matching it.

## Exception hierarchy

All exceptions derive from `bulutklinik::BulutklinikError` (itself a
`` `std::runtime_error` ``):

```
BulutklinikError                  (base — all SDK exceptions)
├── TransportError                (network failure, timeout, DNS, TLS — no HTTP response)
└── ApiError                      (an HTTP response that wasn't a success)
    ├── ValidationError           (422, or errorType "validation")
    ├── AuthenticationError       (401 / logout / failed refresh)
    ├── AuthorizationError        (403)
    ├── NotFoundError             (404)
    └── RateLimitError            (429 — carries retry_after)
```

Each `ApiError` carries `http_status`, `result_type`, `error_type`, `data`, `method`,
`path` and `retry_after` for debugging. `retry_after` (an
`` `std::optional<int>` ``, seconds) is populated from the `Retry-After` header when
present — most relevant on `RateLimitError`.

`ApiError` member fields:

| Field         | Type                          | Notes |
|---------------|-------------------------------|-------|
| `http_status` | `int`                         | The HTTP status code. |
| `result_type` | `std::optional<int>`          | Envelope `resultType`, when parseable. |
| `error_type`  | `nlohmann::json`              | String label or numeric code (raw JSON value). |
| `data`        | `nlohmann::json`              | The raw `data` from the envelope (may be `null`). |
| `method`      | `std::string`                 | Request method. |
| `path`        | `std::string`                 | Request path. |
| `retry_after` | `std::optional<int>`          | Seconds from `Retry-After`, when present. |

The human-readable message is available through `what()` (inherited from
`` `std::runtime_error` ``).

```cpp
#include <bulutklinik/bulutklinik.hpp>
#include <iostream>

try {
    client.payments().pay(input);
} catch (const bulutklinik::RateLimitError& e) {
    if (e.retry_after) {
        std::cerr << "throttled — retry after " << *e.retry_after << "s\n";
    }
} catch (const bulutklinik::ValidationError& e) {
    std::cerr << "invalid request: " << e.data.dump() << "\n";
} catch (const bulutklinik::AuthenticationError& e) {
    std::cerr << "session expired — re-login required\n";
} catch (const bulutklinik::ApiError& e) {
    std::cerr << "API error " << e.http_status << " on "
              << e.method << " " << e.path << ": " << e.what() << "\n";
} catch (const bulutklinik::TransportError& e) {
    std::cerr << "network problem: " << e.what() << "\n";
}
```

> Order your `catch` clauses from most-specific to least-specific: the subtypes
> (`RateLimitError`, `ValidationError`, …) must come before `ApiError`, which in
> turn must come before `BulutklinikError` — otherwise the base handler catches
> everything first.

## Mapping precedence

The SDK picks the most specific exception type in this order:

1. `result_type == 2` (logout) → `AuthenticationError` (and the token store is cleared)
2. string `error_type == "validation"` (case-insensitive) or HTTP 422 → `ValidationError`
3. HTTP status: 401 → `AuthenticationError`, 403 → `AuthorizationError`,
   404 → `NotFoundError`, 429 → `RateLimitError`
4. otherwise (including numeric `error_type`, or 2xx with `result_type != 0`) → `ApiError`

Because `error_type` may be numeric, the SDK guards (`error_type.is_string()`) before
string-matching it — and so should any code that inspects `e.error_type`.
