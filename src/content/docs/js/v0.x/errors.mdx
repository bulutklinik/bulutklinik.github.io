# Errors & responses

## The response envelope

Every API response is a JSON envelope. The SDK unwraps it for you — on success it
returns the `data` payload directly; on failure it throws a typed error.

```jsonc
{
  "resultType": 0,            // state code (see below)
  "errorType": "validation",  // string label OR numeric code
  "errorMessage": "…",        // human-readable, localized via `lang`
  "successMessage": "…",
  "data": { /* payload */ }   // what the SDK returns on success
}
```

A call is successful only when the HTTP status is 2xx **and** `resultType === 0`.
The `resultType` state codes are exported as `ResultType`:

| Value | Name      | Meaning |
|:-----:|-----------|---------|
| `0`   | `Success` | Return `data`. |
| `1`   | `Error`   | Throw `ApiError` (or a subtype). |
| `2`   | `Logout`  | Session revoked — token store cleared, `AuthenticationError`. |
| `3`   | `Update`  | Client/app too old — `ApiError`. |
| `4`   | `Refresh` | Token expired — triggers silent refresh + retry. |

> `errorType` is **polymorphic**: some endpoints return a string label
> (`"validation"`), others a numeric code. The SDK treats it as a refinement hint
> only when it is a string.

## Error hierarchy

All errors extend `BulutklinikError`:

```
BulutklinikError                  (base — all SDK errors)
├── TransportError                (network failure, timeout, DNS, TLS — no HTTP response)
└── ApiError                      (an HTTP response that wasn't a success)
    ├── ValidationError           (422, or errorType "validation")
    ├── AuthenticationError       (401 / logout / failed refresh)
    ├── AuthorizationError        (403)
    ├── NotFoundError             (404)
    └── RateLimitError            (429 — carries retryAfter)
```

Each `ApiError` carries `httpStatus`, `resultType`, `errorType`, `data`, `method`
and `path` for debugging. `RateLimitError` adds `retryAfter` (seconds, when the
`Retry-After` header is present).

```ts
import {
  ApiError,
  AuthenticationError,
  RateLimitError,
  ValidationError,
  TransportError,
} from "@bulutklinik/sdk";

try {
  await client.payments.pay(input);
} catch (err) {
  if (err instanceof RateLimitError) {
    console.warn(`throttled — retry after ${err.retryAfter}s`);
  } else if (err instanceof ValidationError) {
    console.warn("invalid request", err.data);
  } else if (err instanceof AuthenticationError) {
    console.warn("session expired — re-login required");
  } else if (err instanceof TransportError) {
    console.error("network problem", err.cause);
  } else if (err instanceof ApiError) {
    console.error(`API error ${err.httpStatus} on ${err.method} ${err.path}`);
  }
}
```

## Mapping precedence

The SDK picks the most specific error type in this order:

1. `resultType === 2` (logout) → `AuthenticationError`
2. string `errorType === "validation"` (or HTTP 422) → `ValidationError`
3. HTTP status: 401 → `AuthenticationError`, 403 → `AuthorizationError`,
   404 → `NotFoundError`, 429 → `RateLimitError`
4. otherwise (including numeric `errorType`, or 2xx with `resultType !== 0`) → `ApiError`

Because `errorType` may be numeric, the SDK guards before string-matching it — and
so should any code that inspects `err.errorType`.
